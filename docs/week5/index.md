# 인덱스(Index) 학습 정리

---

## Q1. 인덱스라는 것에 대해서 공부해보고 싶어. 막연히 실무에서 어떤 역할을 한다 알고 쓰고는 있지만 그 안의 원리 같은 것들은 내가 모르거든.. 근데 내가 이해력이 약하고 깊은 구조에 대해서는 잘 모르기 대문에 너가 설명을 좀 이해하기 쉽게 해줘봐.

### A1.

# 인덱스(Index) 쉽게 이해하기

## 1. 비유로 시작하자

**책의 목차**를 떠올려보세요.

500페이지짜리 책에서 "트랜잭션"이라는 단어를 찾고 싶다면:
- **목차 없이**: 1페이지부터 500페이지까지 한 장씩 넘기며 찾아야 함 → 느림
- **목차 있으면**: "트랜잭션 → 237페이지" 바로 펼침 → 빠름

**DB 인덱스가 바로 이 "목차" 역할**입니다.

---

## 2. 인덱스가 없으면 어떻게 될까?

```
SELECT * FROM product WHERE name = '나이키 운동화';
```

인덱스가 없으면 DB는 product 테이블의 **모든 행을 처음부터 끝까지** 읽습니다.
이걸 **Full Table Scan**이라고 합니다.

```
행 1: 아디다스 모자      → 아닌데? 다음
행 2: 구찌 가방          → 아닌데? 다음
행 3: 나이키 운동화      → 찾았다! 근데 뒤에 더 있을 수도 있으니...
행 4: 뉴발란스 신발      → 아닌데? 다음
...
행 100만: 프라다 지갑    → 끝까지 다 봤다
```

100만 건이면 100만 번 비교합니다. 느릴 수밖에 없죠.

---

## 3. 인덱스가 있으면?

`name` 컬럼에 인덱스를 만들면, DB는 **정렬된 별도의 자료구조**를 만듭니다.

이 자료구조가 바로 **B-Tree(B트리)** 입니다.

---

## 4. B-Tree를 쉽게 이해하기

어려운 이름이지만, 원리는 단순합니다. **"반씩 쪼개서 찾기"** 입니다.

### 숫자 맞추기 게임으로 이해하기

친구가 1~100 사이의 숫자 하나를 생각했습니다. 어떻게 찾을까요?

```
나: "50보다 커?"     → "응"
나: "75보다 커?"     → "아니"
나: "62보다 커?"     → "응"
나: "68보다 커?"     → "아니"
나: "65보다 커?"     → "아니"
나: "63이야?"        → "응!"
```

100번 물어볼 걸 **6~7번**만에 찾았습니다!

B-Tree가 바로 이 원리입니다.

### B-Tree 구조 그림

```
                        [50]                    ← 루트 노드
                       /    \
                  [25]        [75]              ← 중간 노드
                 /    \      /    \
             [10,20] [30,40] [60,70] [80,90]    ← 리프 노드
                                                   (실제 데이터 위치 포인터)
```

"63"을 찾는다면:
1. 루트: 63 > 50 → 오른쪽
2. 중간: 63 < 75 → 왼쪽
3. 리프: [60,70] 범위에서 63 찾기

**3단계만에 도착!** 100만 건이어도 대략 **20단계** 정도면 찾습니다.

> 수학적으로: log₂(1,000,000) ≈ 20

---

## 5. 실제 DB에서는 어떻게 생겼나?

MySQL(InnoDB)의 인덱스는 크게 두 종류입니다.

### (1) 클러스터드 인덱스 (Clustered Index)

- **테이블당 딱 1개** (보통 Primary Key)
- **실제 데이터가 이 순서대로 물리적으로 저장됨**
- 비유: 사전 — 단어 자체가 가나다순으로 정렬되어 있음

```
PK(id)로 정렬된 실제 데이터:
┌────┬──────────────┬────────┐
│ id │ name         │ price  │
├────┼──────────────┼────────┤
│  1 │ 아디다스 모자  │ 25000  │
│  2 │ 구찌 가방     │ 150000 │
│  3 │ 나이키 운동화  │ 89000  │
│  4 │ 뉴발란스 신발  │ 79000  │
└────┴──────────────┴────────┘
```

`WHERE id = 3` → B-Tree 타고 바로 데이터에 도착

### (2) 세컨더리 인덱스 (Secondary Index)

- **여러 개 만들 수 있음**
- 인덱스에는 **컬럼값 + PK값**이 저장됨
- 비유: 책 뒤의 "찾아보기" — 키워드 → 페이지 번호

```
name 인덱스 (정렬됨):
┌──────────────┬────────┐
│ name         │ PK(id) │
├──────────────┼────────┤
│ 구찌 가방     │   2    │
│ 나이키 운동화  │   3    │  ← 여기서 PK=3을 얻고
│ 뉴발란스 신발  │   4    │
│ 아디다스 모자  │   1    │
└──────────────┴────────┘
        ↓
  PK=3으로 클러스터드 인덱스에서 실제 데이터 가져옴
```

이 **두 번 찾는 과정**을 "테이블 액세스" 또는 "랜덤 I/O"라고 합니다.

---

## 6. 그래서 인덱스를 무조건 많이 만들면 좋은 거야?

**아닙니다!** 인덱스의 트레이드오프가 있습니다.

| | 장점 | 단점 |
|---|---|---|
| **SELECT** | 빨라짐 | - |
| **INSERT** | - | 느려짐 (인덱스에도 추가해야 하니까) |
| **UPDATE** | - | 느려짐 (인덱스 값도 바꿔야 하니까) |
| **DELETE** | - | 느려짐 (인덱스에서도 지워야 하니까) |
| **저장공간** | - | 인덱스도 디스크 공간을 차지함 |

비유하면: 책에 목차를 10개 만들면 찾기는 편하지만, **책 내용이 바뀔 때마다 목차 10개를 전부 수정**해야 합니다.

---

## 7. 실무에서 자주 쓰는 패턴

### 어디에 인덱스를 걸어야 할까?

```sql
-- (1) WHERE 조건에 자주 쓰는 컬럼
SELECT * FROM orders WHERE user_id = 123;
-- → user_id에 인덱스

-- (2) JOIN 조건 컬럼
SELECT * FROM orders o JOIN products p ON o.product_id = p.id;
-- → product_id에 인덱스

-- (3) ORDER BY / GROUP BY 컬럼
SELECT * FROM products ORDER BY created_at DESC;
-- → created_at에 인덱스
```

### 복합 인덱스 (여러 컬럼을 합친 인덱스)

```sql
-- 이런 쿼리가 자주 나온다면:
SELECT * FROM orders WHERE user_id = 123 AND status = 'PAID';

-- 복합 인덱스를 만든다:
CREATE INDEX idx_orders_user_status ON orders(user_id, status);
```

여기서 중요한 규칙이 **"왼쪽부터 매칭"** 입니다.

```
인덱스: (user_id, status)

WHERE user_id = 123                    → 인덱스 사용 O
WHERE user_id = 123 AND status = 'PAID' → 인덱스 사용 O
WHERE status = 'PAID'                  → 인덱스 사용 X (왼쪽이 빠졌으니까!)
```

전화번호부와 같습니다. 성(姓) → 이름 순으로 정렬되어 있는데, 이름만으로는 찾을 수 없는 것과 같습니다.

---

## 8. 한 줄 요약

> **인덱스 = 정렬된 목차(B-Tree)**
> 읽기는 빨라지고, 쓰기는 느려진다.
> 자주 조회하는 컬럼에만 필요한 만큼 만들어라.

## Q2. B-Tree이진 트리를 적용한다면 인덱스를 걸지 않았을 때랑 걸었을 때랑 속도 차이가 2배정도 나려나?

### A2.

# 2배가 아니라, 차이가 어마어마합니다

## 직접 비교해봅시다

데이터가 **100만 건**이라고 가정할게요.

| 방식 | 비교 횟수 | 비유 |
|------|----------|------|
| **인덱스 없음** (Full Scan) | **1,000,000번** | 1페이지부터 끝까지 넘기기 |
| **인덱스 있음** (B-Tree) | **약 20번** | 반씩 쪼개서 찾기 |

**약 5만 배** 차이입니다. 2배가 아니에요!

---

## 왜 이렇게 차이가 나는지 직관적으로 보면

"반씩 쪼개기"가 얼마나 강력한지 직접 세어볼게요.

```
데이터 1,000,000건에서 하나를 찾을 때:

1번째 비교: 1,000,000 → 500,000  (절반 제거)
2번째 비교:   500,000 → 250,000
3번째 비교:   250,000 → 125,000
4번째 비교:   125,000 →  62,500
5번째 비교:    62,500 →  31,250
6번째 비교:    31,250 →  15,625
7번째 비교:    15,625 →   7,812
8번째 비교:     7,812 →   3,906
9번째 비교:     3,906 →   1,953
10번째 비교:    1,953 →     976
11번째 비교:      976 →     488
12번째 비교:      488 →     244
13번째 비교:      244 →     122
14번째 비교:      122 →      61
15번째 비교:       61 →      30
16번째 비교:       30 →      15
17번째 비교:       15 →       7
18번째 비교:        7 →       3
19번째 비교:        3 →       1  ← 찾았다!
```

**19~20번이면 끝**입니다.

---

## 데이터가 늘어날수록 격차는 더 벌어집니다

```
데이터 건수        Full Scan    B-Tree     몇 배 차이?
─────────────────────────────────────────────────────
100건              100번        7번        14배
1,000건            1,000번      10번       100배
10,000건           10,000번     14번       714배
100,000건          100,000번    17번       5,882배
1,000,000건        1,000,000번  20번       50,000배
10,000,000건       10,000,000번 23번       434,782배
```

데이터가 **10배** 늘어나도 B-Tree는 비교 횟수가 **3~4번만 추가**됩니다.

이게 **O(N) vs O(log N)** 의 차이예요.

```
시간
 ^
 |  /  ← Full Scan (O(N)): 데이터에 비례해서 직선으로 증가
 | /
 |/___________________  ← B-Tree (O(log N)): 거의 평평하게 유지
 └──────────────────→ 데이터 건수
```

---

## 실제 DB에서는 더 빠릅니다

실제 MySQL의 B-Tree는 이진 트리(자식 2개)가 아니라 **자식이 수백 개**인 트리입니다.

```
이진 트리:  한 노드에서 2갈래      → 높이 20
B-Tree:    한 노드에서 200~300갈래 → 높이 3~4
```

왜냐하면 디스크에서 한 번 읽을 때 **16KB 페이지** 단위로 읽는데, 그 안에 키를 수백 개 담을 수 있거든요. 그래서 실제로는 **3~4번의 디스크 읽기**만으로 수백만 건에서 원하는 데이터를 찾습니다.

---

## 한 줄 요약

> 2배가 아니라 **수만~수십만 배** 차이입니다.
> 데이터가 많아질수록 이 격차는 더 벌어집니다.
> 이게 인덱스가 실무에서 반드시 필요한 이유예요.

## Q3. 실제 MySQL의 B-Tree는 이진 트리(자식 2개)가 아니라 자식이 수백 개인 트리입니다. 이거에 대해서 조금 더 설명해줄래?

### A3.

# B-Tree가 자식이 수백 개인 이유

## 먼저 이진 트리부터 보자

이진 트리는 한 노드에 **값 1개, 자식 2개**입니다.

```
        [50]
       /    \
    [25]    [75]
    /  \    /  \
 [10] [30] [60] [90]
```

값을 찾을 때마다 **한 층씩** 내려가야 하고, 100만 건이면 높이가 약 20층입니다.

---

## 핵심 문제: 디스크는 느리다

여기서 중요한 현실이 있어요.

```
메모리(RAM)에서 데이터 읽기:  ~100 나노초
디스크(SSD)에서 데이터 읽기:  ~100,000 나노초 (약 1,000배 느림)
디스크(HDD)에서 데이터 읽기:  ~10,000,000 나노초 (약 100,000배 느림)
```

DB의 인덱스 데이터는 **디스크에 저장**됩니다. 그래서 트리를 한 층 내려갈 때마다 디스크를 한 번 읽어야 해요.

```
이진 트리 높이 20층 = 디스크 읽기 20번 = 느리다!
```

그래서 DB는 이렇게 생각한 겁니다:

> "한 번 디스크를 읽을 때, 값을 1개만 가져오지 말고 **한꺼번에 많이** 가져오자!"

---

## 디스크는 "페이지" 단위로 읽는다

디스크는 1바이트만 읽으려 해도 **최소 단위(페이지)**를 통째로 읽습니다.

MySQL InnoDB의 페이지 크기는 **16KB**입니다.

```
"1바이트만 필요해도 16KB를 읽어야 한다"

┌─────────────────────────────────┐
│          16KB 페이지              │
│                                 │
│  어차피 통째로 읽는 건데...        │
│  여기를 값으로 꽉 채우면?         │
│                                 │
└─────────────────────────────────┘
```

이진 트리처럼 값 1개만 넣으면 **15.99KB가 낭비**입니다.

그래서 **한 페이지에 값을 수백 개 채우자!** 가 B-Tree의 아이디어입니다.

---

## B-Tree: 한 노드 = 한 페이지(16KB)

```
한 노드(16KB 페이지) 안에:
┌──────────────────────────────────────────────────┐
│  [3] [7] [12] [25] [31] [45] [52] ... [980] [995] │
│  ↓   ↓    ↓    ↓    ↓    ↓    ↓        ↓     ↓   │
│  자  자   자   자   자   자   자       자    자    │
│  식  식   식   식   식   식   식       식    식    │
│  포  포   포   포   포   포   포       포    포    │
│  인  인   인   인   인   인   인       인    인    │
│  터  터   터   터   터   터   터       터    터    │
└──────────────────────────────────────────────────┘

키 1개 크기: 8바이트 (BIGINT 기준)
포인터 1개:  6바이트
한 쌍:      14바이트

16KB ÷ 14바이트 ≈ 약 1,100개
```

대략 **한 노드에 키가 수백~천 개** 들어갑니다.

---

## 높이 비교

이게 높이에 어떤 영향을 주는지 비교해볼게요.

```
데이터 1,000만 건 기준:

이진 트리 (자식 2개):
  높이 = log₂(10,000,000) ≈ 23층
  → 디스크 읽기 23번

B-Tree (자식 1,000개):
  높이 = log₁₀₀₀(10,000,000) ≈ 2.3 → 3층
  → 디스크 읽기 3번
```

실제로 그려보면:

```
이진 트리 (높이 23):
        [1개]
       /     \
    [1개]   [1개]
    /  \    /  \
   ... (23층까지 계속)


B-Tree (높이 3):
              ┌──────── [1,000개의 키] ────────┐        ← 1층 (루트)
              │              │                 │
     [1,000개의 키]   [1,000개의 키]    [1,000개의 키]   ← 2층
      ↓↓↓...↓↓↓       ↓↓↓...↓↓↓        ↓↓↓...↓↓↓
    [데이터] [데이터]  [데이터] [데이터]  [데이터] [데이터] ← 3층 (리프)
```

```
1층(루트):     1 × 1,000 = 1,000개
2층:       1,000 × 1,000 = 1,000,000개
3층(리프): 1,000,000 × 1,000 = 1,000,000,000개 (10억!)
```

**3층만으로 10억 건**을 커버합니다.

---

## 게다가 루트는 메모리에 있다

자주 쓰는 상위 노드는 **메모리에 캐싱**되어 있어요.

```
1층 (루트):  항상 메모리에 있음     → 디스크 읽기 0번
2층:        거의 메모리에 있음      → 디스크 읽기 0번
3층 (리프):  디스크에서 읽음        → 디스크 읽기 1번
```

그래서 실무에서 인덱스를 타면, 수백만 건에서도 **디스크 읽기 1~2번**으로 데이터를 찾습니다.

---

## 한 줄 요약

> 디스크는 어차피 16KB를 통째로 읽으니까,
> 그 안에 키를 수백 개 채워서 **트리 높이를 3~4층으로 줄인 것**이 B-Tree입니다.
> 높이가 낮을수록 디스크 읽기 횟수가 줄고, 그래서 빠릅니다.

---

## Q4. 근데 DB의 인덱스는 왜 메모리가 아니라 디스크에 저장되는거야?

### A4.

핵심은 **메모리(RAM)의 한계** 때문입니다.

---

## 메모리는 비싸고, 작고, 날아간다

| | 메모리(RAM) | 디스크(SSD) |
|---|---|---|
| **가격** | 16GB에 ~5만원 | 1TB(=1,000GB)에 ~10만원 |
| **용량** | 보통 16~64GB | 보통 수백GB ~ 수TB |
| **전원 꺼지면?** | 데이터 **사라짐** | 데이터 **유지됨** |

---

## 구체적으로 생각해보면

실무에서 DB 테이블 하나가 수천만 건이면 데이터가 **수십~수백 GB**입니다.

```
예시: 주문 테이블 5,000만 건
- 테이블 데이터:  약 50GB
- 인덱스 3개:    약 15GB
- 합계:          약 65GB
```

이걸 전부 메모리에 올리려면 **RAM이 65GB 이상** 필요합니다. 테이블이 여러 개면 수백 GB가 되죠.

그런 서버를 구성하면 **비용이 어마어마**합니다. 그리고 서버가 갑자기 꺼지면? 메모리에만 있던 데이터는 **전부 사라집니다**.

---

## 그래서 전략은 이렇습니다

```
┌──────────────────────────────────────────────┐
│  메모리 (RAM) - 작지만 빠름                     │
│                                              │
│  자주 쓰는 것만 올려놓자! (캐시)                 │
│  ┌────────────────────────────┐              │
│  │ Buffer Pool (InnoDB)       │              │
│  │ - 루트/중간 노드 (항상 캐싱)  │              │
│  │ - 최근 조회한 리프 노드      │              │
│  │ - 자주 읽는 데이터 페이지     │              │
│  └────────────────────────────┘              │
├──────────────────────────────────────────────┤
│  디스크 (SSD/HDD) - 크고 느리지만 안전함         │
│                                              │
│  전체 데이터를 영구 보관                         │
│  ┌────────────────────────────┐              │
│  │ 테이블 데이터 전체            │              │
│  │ 인덱스 전체                  │              │
│  │ → 서버 꺼져도 안 사라짐       │              │
│  └────────────────────────────┘              │
└──────────────────────────────────────────────┘
```

MySQL InnoDB는 **Buffer Pool**이라는 메모리 공간을 갖고 있어요. 여기에 자주 쓰는 인덱스 페이지와 데이터 페이지를 **캐싱**해둡니다.

그래서 아까 설명한 것처럼:

```
B-Tree 3층 기준:

1층 (루트):  자주 쓰니까 Buffer Pool에 캐싱됨  → 메모리에서 읽음 (빠름)
2층 (중간):  꽤 자주 쓰니까 대부분 캐싱됨      → 메모리에서 읽음 (빠름)
3층 (리프):  양이 많아서 일부만 캐싱됨          → 디스크에서 읽을 수도 있음
```

---

## 한 줄 요약

> **영구 보관 + 용량 한계** 때문에 디스크에 저장하고,
> 자주 쓰는 부분만 메모리(Buffer Pool)에 캐싱해서 빠르게 읽는 구조입니다.
> "전부 메모리에 두면 좋지만, 현실적으로 불가능하니까 똑똑하게 나눠 쓰는 것"이에요.
