# 너무 쉽게 비관(적락)을 하지 마십시오.

## 들어가며

그동안 동시성 문제는 수만 명의 사용자가 몰리는 '대규모 트래픽' 상황에서만 발생하는 문제라고 생각했었다. 하지만 사용자가 단 2명뿐이라도, 찰나에 동일한 데이터에 접근한다면 동시성 이슈는 반드시 발생한다. 그동안 좋은 성능의 서버와 풍부한 자원들에 의지해 동시성을 고려하지 않았던 과거가 놀라울 뿐이다. ㅠ_ㅠ

이번 과제의 핵심은 이커머스 주문 흐름에 완벽한 동시성 제어를 적용하는 것이었다.
재고 차감부터 쿠폰 사용, 포인트 차감, 그리고 장바구니 삭제까지. 이 모든 과정은 하나의 원자적 단위로 처리되어야 하며, 어떠한 동시 요청에도 데이터의 정합성이 깨져서는 안 된다.

처음에는 단순히 "비관적 락을 걸면 해결되겠지"라고 생각했다.
하지만 도메인마다 데이터의 특성과 충돌 빈도가 달라, 하나의 일괄된 전략만으로는 성능과 안정성을 모두 잡을 수 없었다.

#### 이 글은 동시성이 발생할 수 있는 도메인에 3가지 락 전략을 적용하며 마주했던 내용들에 대한 정리이다.

## 동시성 문제를 해 세 가지 해법

동시성 문제라는 것 은 **여러 스레드가 같은 데이터를 동시에 읽고 쓸 때, 일부 변경이 유실되는 것**(Lost Update)이다.

편의점 알바를 생각해보자. 재고 장부에 "콜라 5개"라고 적혀있다. 알바생 A와 B가 동시에 장부를 보고, 각자 1개를 팔고, 각자 "콜라 4개"라고 고쳐 쓴다. 실제로는 2개가 팔렸는데 장부에는 1개만 빠져있다. 이게 바로 **Read-Decide-Write** 패턴에서 발생하는 Lost Update다.

```java
A: 재고 = 5 읽음 → 충분함 → 재고 = 4로 저장
B: 재고 = 5 읽음 → 충분함 → 재고 = 4로 저장  ← A의 차감이 사라짐
```

이 문제를 방지하는 세 가지 전략이 있다.

### 비관적 락 (Pessimistic Lock)

비관적 락은 **화장실 문 잠그기**와 같다. 들어가면서 문을 잠그고, 볼 일을 다 보고 나올 때까지 다른 사람은 문 앞에서 기다려야 한다. 데이터를 읽는 시점에 `SELECT ... FOR UPDATE`로 행 잠금을 걸어, 트랜잭션이 끝날 때까지 다른 트랜잭션의 접근을 막는다. 줄을 서서 기다리는 만큼 느려질 수 있고, 서로 다른 화장실 문을 잠근 채 상대방 것을 기다리면 데드락에 빠진다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT po FROM ProductOption po WHERE po.id = :id")
Optional<ProductOption> findByIdWithLock(@Param("id") Long id);
```

### 낙관적 락 (Optimistic Lock)

낙관적 락은 **구글 문서 동시 편집**에 가깝다. 일단 자유롭게 수정하되, 저장하는 순간 "내가 읽은 이후에 누가 바꿨으면 내 저장은 취소"하는 방식이다. JPA의 `@Version`을 사용하면 UPDATE 시 `WHERE version = ?` 조건이 자동으로 붙는다. 누군가 먼저 version을 올렸다면 내 UPDATE는 실패하고, 트랜잭션 전체가 롤백된다. 줄을 서지 않아 빠르지만, 충돌하면 했던 일을 전부 되돌려야 하는 대가가 따른다.

```java
@Version
private Long version;
```

### 원자적 업데이트 (Atomic UPDATE)

원자적 업데이트는 **장부를 읽지 않고 바로 고치는** 것과 같다. "현재 값이 뭐든 +1 해줘"라고 DB에게 맡기면, 읽기 자체가 없으니 Lost Update가 발생할 수 없다. 가장 단순하고 빠르며, WHERE 절에 조건을 추가하면 조건부 연산도 가능하다. 예를 들어 `UPDATE point SET balance = balance - :amount WHERE id = :id AND balance >= :amount`처럼 작성하면, 잔액이 충분할 때만 차감이 수행되고, affected rows가 0이면 잔액 부족으로 판단할 수 있다. 다만 읽고 → 판단하고 → 쓰는 과정이 필요한 경우, 원자적 업데이트로는 그 사이에 애플리케이션 로직을 끼워넣을 수 없다는 한계가 있다.

```java
@Modifying
@Query("UPDATE Product p SET p.likeCount = p.likeCount + 1 WHERE p.id = :productId")
void incrementLikeCount(@Param("productId") Long productId);
```

그러면 어떤 도메인에 어떤 전략을 적용할까? 세 가지 질문으로 판단했다:

1. **충돌이 얼마나 자주 발생하는가?**
2. **충돌 시 실패해도 되는가, 대기해서라도 성공해야 하는가?**
3. **비즈니스 검증이 필요한가, 단순 연산인가?**

## 도메인별 적용 사례

### [좋아요] 좋아요는 가볍게, UPDATE는 원자적으로

좋아요는 단순 카운터 증감이다. 조건 검증 없이 +1/-1만 하면 되고, 모든 요청이 성공해야 한다. 좋아요를 눌렀는데 "다시 시도해주세요"가 뜨면 사용자 입장에서는 당황스러울 뿐이다.

좋아요 카운터는 값을 읽어서 판단한 뒤 쓰는 Read-Modify-Write 패턴이 아니다. 읽을 필요가 없으니 락을 걸 이유도 없다. 그래서 장부를 아예 읽지 않는 원자적 UPDATE를 택했다.
```java
@Modifying
@Query("UPDATE Product p SET p.likeCount = p.likeCount + 1 WHERE p.id = :productId")
void incrementLikeCount(@Param("productId") Long productId);
```

DB가 `likeCount = likeCount + 1`을 원자적으로 실행하므로 Read-Modify-Write 패턴 자체가 사라진다. UPDATE 자체가 실행시 조건의 대상이 되는 행을 락(잠금)하므로 10개 스레드가 동시에 호출해도 정확히 10이 된다.
추가로 `Like` 엔티티에 `UniqueConstraint`를 걸어 동일 사용자의 중복 좋아요도 DB 레벨에서 차단했다.

### [쿠폰 사용] 한 번 쓰면 끝인 쿠폰, 낙관(적락)으로 처리하자!

쿠폰의 상태를 사용처리 하는 것은 한 번 쓰면 끝이고, 되돌릴 수 없다. 같은 쿠폰으로 동시 주문하는 상황은 내가 두 대의 폰으로 동시에 결제 버튼을 누르지 않는 이상 현실적으로 거의 발생하지 않는다. 충돌이 나더라도 재시도할 의미가 없다. 이미 누군가 써버렸으니까.

그래서 **줄 서서 기다릴 필요 없이, 충돌이 나면 바로 "이미 사용된 쿠폰입니다"**를 던지는 낙관적 락을 선택했다.

```java
@Version
private Long version;
```

`@Version`을 추가하면 JPA가 `UPDATE ... WHERE version = ?`을 자동 수행한다.
먼저 커밋한 하나만 성공하고, 나머지는 `OptimisticLockingFailureException`으로 빠르게 실패한다.

예외 처리는 **2단계 구조**로 설계했다.
대부분은 `isAvailable()` 검증에서 걸러지고, 극소수의 동시 경합만 `GlobalExceptionHandler`에서 409 Conflict로 처리한다. `saveAndFlush()`를 쓰면 영속성 컨텍스트 전체가 flush되는 부작용이 있어 `save()` + 커밋 시점 감지 방식을 택했다.

### [재고 차감] 재고는 비관(적락)해야 안전하다

인기 상품의 재고는 **한정판 콘서트 티켓**과 같다. 수십 명이 동시에 달려들고, 없는 걸 팔면(초과 판매) 큰 사고다. "재고가 충분한데 동시 요청 때문에 결제 실패"가 뜨면? 고객은 떠나고 매출은 사라진다.

여기서는 Atomic UPDATE를 쓸 수 없다. "재고가 충분한지 확인한 후 차감"이라는 조건부 연산이 필요하기 때문이다.

낙관적 락은 충돌 시 실패하는데, 재고가 남아있는데 실패하면 안 된다.
결국 **화장실 문을 잠그고 한 명씩 들여보내는** 비관적 락을 선택했다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT po FROM ProductOption po WHERE po.id = :id AND po.deletedAt IS NULL")
Optional<ProductOption> findByIdWithLockAndDeletedAtIsNull(@Param("id") Long id);
```

여러 옵션에 락을 걸 때는 **optionId 오름차순 정렬**로 데드락을 방지했다.
A가 1번 화장실 → 2번 화장실 순으로 잠그고, B가 2번 → 1번 순으로 잠그면 서로 기다리다 영원히 멈춘다. 항상 번호가 작은 화장실부터 잠그면 이런 교착 상태가 생기지 않는다.

### [포인트] 포인트도 현금이니까??? 비관적으로 가자...

처음에는 쿠폰과 비슷하니까 낙관적 락을 고려했다. 하지만 결정적 차이가 있었다.
이 차이를 이해하려면 먼저 **두 락 전략이 실패했을 때 어떤 일이 벌어지는지**를 알아야 한다.

#### 비관적 락 vs 낙관적 락의 롤백 시점

**비관적 락**은 화장실 문을 잠그는 것이니, 문 앞에서 기다리다 들어가면 항상 최신 상태를 본다.
잔액이 충분하면 성공, 부족하면 비즈니스 검증에서 예외가 발생할 뿐이다. **롤백 자체가 일어나지 않는다.**

```java
A: SELECT FOR UPDATE (balance=1000) → 락 획득 → use(800) → balance=200 → 커밋 → 락 해제
B: SELECT FOR UPDATE → 대기... → 락 획득 (balance=200) → use(800) → 잔액 부족 예외!
```

**낙관적 락**은 일단 자유롭게 진행하되, 커밋 순간 version이 바뀌어 있으면 **트랜잭션 전체를 롤백**한다.
핵심은 "전체"라는 단어다 — 포인트뿐 아니라 **같은 트랜잭션에서 수행한 모든 변경이 함께 날아간다.**

```java
A: balance=1000 읽음 (version=0) → use(800) → UPDATE WHERE version=0 → 성공! version=1
B: balance=1000 읽음 (version=0) → use(800) → UPDATE WHERE version=0 → 실패! 트랜잭션 전체 롤백
```

#### 주문 트랜잭션에서 낙관적 락이 위험한 이유

이 차이가 결정적이었다.
포인트 사용은 `OrderFacade.createOrder()`의 `@Transactional` 안에서 재고 차감, 쿠폰 사용 **이후**에 실행된다. 비유하자면, 콘서트 티켓을 끊고(재고), 할인 쿠폰을 찢고(쿠폰), 마지막으로 포인트를 쓰려는데 여기서 낙관적 락이 터지면 — **티켓도 돌려놓고, 찢은 쿠폰도 다시 붙여야 한다.** 재시도하려 해도 이미 티켓이 매진됐거나 쿠폰 상태가 바뀌어 있을 수 있다.

```java
createOrder() @Transactional {
    1. 재고 차감 (비관적 락) -> 성공
    2. 쿠폰 사용 (낙관적 락) -> 성공
    3. 포인트 차감 ← 만약 여기서 낙관적 락 실패하면?
    → OptimisticLockingFailureException → 트랜잭션 전체 롤백
    → 1번 재고 차감도 롤백, 2번 쿠폰 사용도 롤백
}
```

**포인트가 충분한데 주문이 실패하는 건 납득하기 어렵다.**
반면 비관적 락은 순차 대기 후 처리하므로, 잔액만 충분하면 두 주문 모두 성공할 수 있다.
게다가 포인트는 **회원당 1 row**(`member_id` unique)이므로, 비관적 락이 걸려도 해당 회원만 잠기고 다른 회원은 영향이 없다. 비관적 락의 단점인 "줄 서기"가 사실상 발생하지 않는 구조였다.

### [쿠폰 수정,삭제 / 장바구니]

**쿠폰 수정과 삭제**는 어드민과 고객이 동시에 같은 쿠폰에 손을 대는 경우다.
어드민이 쿠폰을 삭제하는 찰나에 고객이 다운로드하면, 이미 사라진 쿠폰이 발급된다. 빈도는 낮지만 발생하면 문제가 되므로, 수정,삭제 시에만 비관적 락을 적용하고 일반 조회는 락 없이 분리했다.

**장바구니**는 고정관념을 하나 버려야 했다.
기존에는 Soft Delete를 쓰고 있었는데, UPSERT(`INSERT ... ON DUPLICATE KEY UPDATE`)를 적용하려면 유니크 제약이 필요했고, MySQL은 partial unique index를 지원하지 않아 삭제된 row와 충돌이 났다. 곰곰이 생각해보니 장바구니는 주문 후 사라지는 임시 데이터다. 삭제 이력을 보존할 이유가 없었다. 과감하게 **Hard Delete로 전환**하고 UPSERT로 Read-then-Write 패턴을 완전히 제거했다.

## 가장 오래 삽질한 문제: JPA 1차 캐시의 함정

이번 과제에서 가장 멘붕이 왔던 순간이다.
비관적 락을 정확히 걸었는데 동시성 테스트가 실패했다.
재고 5개에 10명이 주문하면 5명만 성공해야 하는데, **10명 전원이 성공**했다.
락이 아예 안 걸린 것처럼 동작한 것이다. "내가 뭘 잘못한 거지..?" 한참을 들여다봤다.


```java
Product product = productService.findById(request.productId());       // (1) 일반 조회 -> ProductOption이 1차 캐시에 로드
ProductOption option = productService.deductStock(optionId, quantity); // (2) 비관적 락 조회
```

`(1)`에서 Product를 조회할 때 연관된 ProductOption이 JPA 1차 캐시에 올라간다. `(2)`에서 `SELECT ... FOR UPDATE`를 실행하면 **DB 문은 잠겼는데, JPA는 이미 캐시에 있는 옛날 데이터를 그대로 쓴다.** 락은 DB가 걸고, 데이터 읽기는 JPA가 하는데, 이 둘이 따로 논다는 걸 이때 처음 알았다.

> **비관적 락은 DB 행 잠금이지, JPA 1차 캐시를 갱신하는 메커니즘이 아니다.**

### 1차 시도: `refresh()`로 강제 갱신 — "일단은 되긴 하는데..."

처음 떠올린 해결책은 `entityManager.refresh(option, PESSIMISTIC_WRITE)`였다. "캐시가 문제면 캐시를 강제로 새로고침하면 되지 않나?" 테스트를 돌려보니 통과했다. 잠깐 안도했다.

하지만 알아보니 `refresh()`에는 함정이 있었다. 해당 엔티티에 아직 DB에 반영 안 된 변경(dirty state)이 있으면 그 변경을 날려버린다. 지금은 안전하지만, 나중에 누군가 같은 엔티티를 수정한 뒤 `refresh()`를 호출하면 수정이 조용히 사라지는 시한폭탄이 된다. 무엇보다 **근본 원인 — ProductOption이 1차 캐시에 미리 로드되는 것 — 을 해결하지 않고 반창고만 붙이는 느낌**이라 찜찜했다.

### 최종 해결: 애초에 1차 캐시에 올리지 않기

다시 원인으로 돌아갔다. `findById()`가 ProductOption을 1차 캐시에 로드하는 게 문제라면, **애초에 로드하지 않으면 된다.** 주문 스냅샷에 필요한 건 Product의 `name`, `price`뿐이고, 옵션 정보는 `deductStock()`이 반환하는 값을 쓰면 된다. Product만 조회하는 `findProductOnly()` 메서드를 만들어 1차 캐시 오염을 원천 차단했다.

```java
// 수정 전 — findById()가 ProductOption을 1차 캐시에 로드 → deductStock()에서 stale read
Product product = productService.findById(request.productId());

// 수정 후 — findProductOnly()는 옵션을 로드하지 않음 → deductStock()에서 DB 직접 조회
Product product = productService.findProductOnly(request.productId()); // 옵션 미로드
ProductOption option = productService.deductStock(optionId, quantity);  // 1차 캐시에 없으므로 DB에서 최신값 로드 + 락
```

원인을 제거하니, 테스트가 깔끔하게 통과했다. `refresh()`로 해결했으면 아마 한참 뒤에 더 큰 버그로 다시 만났을 것이다.

## 마치며

"동시성 제어 = 비관적 락"이라는 단순한 공식으로 시작했지만, 결국 도메인마다 다른 전략을 적용하게 됐다.

가장 중요한 판단 기준은 락 기술이 아니라 **"충돌이 얼마나 자주 발생하는가"와 "충돌 시 실패해도 되는가"** 였다. 같은 "잔액 차감"이라도 쿠폰(낙관적 락)과 포인트(비관적 락)에서 다른 전략을 선택한 이유가 여기에 있다.

그리고 가장 고생했던 부분!  **비관적 락은 DB 행 잠금이고, 1차 캐시는 JPA의 동작이다.**
이 둘이 각자의 세계에서 독립적으로 동작한다는 걸 몰라서, 락을 정확히 걸어놓고도 Lost Update를 겪었다. 문을 잠갔는데 열쇠가 안 맞는 느낌이랄까. 동시성 문제를 다룰 때는 프레임워크의 내부 동작까지 이해해야 한다는 걸 알게 되었다.

또한 이번 글에서 다룬 전략들은 모두 DB 레벨의 동시성 제어다.
하지만 애플리케이션 레벨에서도 `synchronized`, `ReentrantLock`, Redis 분산 락 같은 다양한 기법들이 존재한다. 단일 서버를 넘어 다중 인스턴스 환경에서는 DB 락만으로 해결할 수 없는 문제들이 생길 텐데, 이 부분도 추후에 깊이 있게 공부해보고 싶다.

여전히 고민인 부분도 있다.
현재 `createOrder()`는 하나의 `@Transactional` 안에서 쿠폰 검증, 재고 차감, 주문 생성, 쿠폰 사용, 포인트 차감, 장바구니 삭제를 모두 수행한다.
트랜잭션 범위가 넓다 보니 비관적 락의 보유 시간이 길어지고, 중간에 하나라도 실패하면 전부 롤백된다. 순서를 최적화하고 검증을 앞으로 빼는 것만으로는 한계가 있다.
트랜잭션을 분리하면 원자성이 깨지고, 합치면 락 보유 시간이 늘어나는 이 트레이드오프를 어떻게 풀어야 할지는 아직 명쾌한 답을 찾지 못했다.
앞으로 시야가 더 넓어지면 방법울 찾을 수 있겠지..?있겠지..???
