# WIL - 2주차 - 요구 사항 정의 / 시퀀스 다이어그램 / 클래스 다이어그램 / ERD 다이어그램)

## 🧠 이번 주에 새로 배운 것

- 요구사항 문서를 시퀀스 다이어그램 → 클래스 다이어그램 → ERD 순서로 설계하면서, "코드를 치기 전에 설계를 먼저 해야 하는 이유" 를 체감했다. 애매한 요구사항이 50개 가까이 나왔는데, 코딩하면서 이걸 발견했으면 되돌리는 비용이 훨씬 컸을 것 같다.
- 물리적 FK 제약조건을 걸지 않는 실무 관행과 그 이유를 이해하게 됐다. 쓰기 성능, 운영 유연성, DB 분리 대비, Soft Delete 호환까지 여러 관점에서 물리 FK가 오히려 발목을 잡을 수 있다는 걸 알게 됐다.
- 인증(Authentication)은 비즈니스 로직이 아니라 인프라적 관심사라는 개념이 와닿았다. 처음엔 Facade에서 매번 `authenticate()`를 호출하는 구조로 설계했는데, Interceptor로 옮기기로 결정하니 Facade가 깔끔해졌다. 관심사를 분리한다는 게 이런 거구나 싶었다.
- 도메인 모델과 JPA Entity를 분리하는 것의 의미를 OrderItem 설계를 통해 구체적으로 이해했다. Domain의 OrderItem에는 orderId가 없지만 Entity에서는 `@ManyToOne`으로 order를 참조한다. 도메인 모델은 JPA에 의존하지 않는 순수 객체로 유지하고, `order_id` 매핑은 인프라 레이어의 책임이라는 원칙이 명확해졌다.

## 💭 이런 고민이 있었어요

- 좋아요 수를 어떻게 설계 할지 고민이 많았다. Like 테이블에서 매번 COUNT를 집계하면 쿼리 비용이 크고, Product에 `likeCount`를 비정규화하면 정합성 걱정이 생긴다. 결국 비정규화를 선택했는데, 동시성 이슈(동시에 좋아요가 몰리면?)에 대해서는 비관적 락이나 별도 집계 방식을 나중에 검토해야 한다는 숙제가 남았다.
- Application Layer에 Service가 필요한가? 라는 질문이 계속 머릿속을 맴돌았다. OrderService가 ProductRepository, OrderRepository, CartRepository를 다 들고 있어서 "이건 도메인 로직이 아니라 유스케이스 오케스트레이션 아닌가?" 싶었다. 결론적으로 현재 규모에서는 Facade + Domain Service 2단 구조로 충분하다고 판단했지만, 나중에 도메인이 커지면 분리해야 할 시점을 놓치지 않아야겠다고 생각했다.
- 주문 트랜잭션 범위가 너무 넓은 것 같았다. 재고 차감 → 주문 저장 → 장바구니 삭제가 하나의 트랜잭션인데, 트래픽이 몰리면 lock 범위가 넓어져서 병목이 될 수 있다. 일단은 단일 트랜잭션으로 가되, 이벤트 기반 분리를 언제 해야 할지 감을 잡아둬야겠다.
- 시퀀스 다이어그램에 메서드명으로 적었다가 한글로 바꾼 것도 의미 있었다. 처음엔 `getBrand(brandId)`, `authenticate(loginId, password)` 같이 메서드 시그니처 그대로 써넣었는데, 설계 문서는 개발자뿐 아니라 누구나 읽을 수 있어야 한다는 피드백을 받고 "브랜드 조회 요청", "회원 인증 요청"으로 바꿨다. 설계 문서와 코드의 목적이 다르다는 걸 깨달았다.

## 💡 앞으로 실무에 써먹을 수 있을 것 같은 포인트

- 요구사항의 상태코드나 에러 처리가 이상하면 왜 이상한지 근거를 들어서 대안을 제시하는 습관. "요구사항에 500이라고 했으니까 500"이 아니라, HTTP 의미론을 기준으로 설득하는 게 개발자의 역할이라는 걸 배웠다.
- `findByIdAndMemberId(orderId, memberId)` 처럼 한 번의 쿼리로 권한 검증과 조회를 동시에 처리**하는 패턴. 타인에게 주문 존재 여부조차 노출하지 않으면서 쿼리도 하나로 끝난다.
- Audit 컬럼(`created_by`, `updated_by`, `deleted_by`)은 처음부터 넣어두는 게 맞다. 운영 환경에서 "누가 이 데이터를 건드렸지?" 라는 질문에 답하려면 반드시 필요한 정보다.
- 검색 조건 객체(`ProductSearchCondition`, `AdminProductSearchCondition`)로 파라미터를 묶는 패턴. 파라미터가 3개 이상이면 개별 전달보다 객체로 묶는 게 시그니처도 깔끔하고 확장에도 유리하다.
- 인증 처리는 Interceptor + ArgumentResolver에서 공통 처리하고, Facade/Service는 인증된 객체를 받기만 하는 구조. 인증 코드 중복을 없애고 비즈니스 로직에 집중할 수 있게 해준다.

## 🤔 아쉬웠던 점 & 다음 주에 해보고 싶은 것

- 설계 의사결정을 50개 가까이 내리면서 "이 결정이 구현할 때 정말 문제없을까?" 라는 확신이 부족한 부분이 있다. 특히 주문 트랜잭션 범위, like_count 비정규화 정합성, 서비스 간 의존관계 같은 부분은 직접 구현하고 테스트해봐야 감이 잡힐 것 같다.
- 시퀀스 다이어그램에 파라미터 형식 검증(@Valid, Bean Validation)은 반영하지 않기로 했는데, 막상 구현할 때 검증 범위나 에러 메시지를 설계에서 미리 잡아뒀으면 더 수월했을 것 같다는 생각도 든다.
- 다음 주에는 이번 설계를 기반으로 TDD 방식(Red → Green → Refactor)으로 실제 구현을 시작해보고 싶다. 설계할 때 예상하지 못한 문제가 분명 나올 텐데, 그걸 테스트로 먼저 잡아내는 경험을 해보고 싶다.
- 설계 리뷰를 통해 인증 처리 위치를 바꾸는 등 큰 변경이 있었는데, 처음부터 횡단 관심사를 분리하는 관점으로 설계했다면 리뷰 후 수정이 줄었을 것 같다. 다음엔 설계 초기에 "이건 비즈니스 로직인가, 인프라 관심사인가?"를 먼저 따져보는 습관을 들여야겠다.
