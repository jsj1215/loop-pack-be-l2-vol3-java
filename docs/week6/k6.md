# K6 부하테스트 - 결제 시스템 Resilience 검증

## 개요

PG Simulator와 연동된 결제 API에 대해 k6 부하테스트를 수행하여,
Timeout / CircuitBreaker / Fallback / Retry 4가지 Resilience 전략이 정상 동작하는지 검증합니다.

## 테스트 대상

| 항목 | 값 |
|------|-----|
| Commerce API | `http://localhost:8080` |
| PG Simulator | `http://localhost:8082` |
| 결제 API | `POST /api/v1/payments` (Manual), `POST /api/v2/payments` (Resilience4j) |
| 콜백 API | `POST /api/v1/payments/callback` |
| 검증 API | `GET /api/v1/payments/verify?orderId={id}` |

## PG Simulator 특성

| 항목 | 값 |
|------|-----|
| 요청 성공 확률 | 60% |
| 요청 지연 | 100ms ~ 500ms |
| 처리 지연 | 1s ~ 5s |
| 처리 결과 - 성공 | 70% |
| 처리 결과 - 한도 초과 | 20% |
| 처리 결과 - 잘못된 카드 | 10% |

## 현재 시스템 설정

### RestTemplate Timeout
```yaml
pg:
  timeout:
    connect: 1000   # 연결 타임아웃 1초
    read: 3000      # 읽기 타임아웃 3초
```

### Resilience4j CircuitBreaker (pgCircuit)
```yaml
resilience4j:
  circuitbreaker:
    instances:
      pgCircuit:
        sliding-window-size: 10
        failure-rate-threshold: 50        # 실패율 50% 초과 시 Open
        wait-duration-in-open-state: 10s  # Open 유지 10초
        permitted-number-of-calls-in-half-open-state: 2
        slow-call-duration-threshold: 2s
        slow-call-rate-threshold: 50
```

### Manual CircuitBreaker (V1)
- Sliding Window Size: 10
- Failure Rate Threshold: 50%
- Open State Duration: 10초
- Half-Open Permitted Calls: 2
- Retry: 최대 2회, 500ms 간격

## 테스트 시나리오

### Scenario 1: Timeout 검증

**목적**: PG 응답 지연 시 설정된 타임아웃(read: 3s)이 정상 동작하는지 확인

**검증 포인트**:
- 3초 이내 응답이 오는 경우 정상 처리되는지
- 3초 초과 시 타임아웃으로 실패 처리되는지
- 타임아웃 발생 시 Payment 상태가 UNKNOWN으로 마킹되는지
- 타임아웃이 발생해도 스레드가 무한 대기하지 않는지

**시나리오**: 점진적으로 동시 사용자를 늘려 PG 응답 지연을 유발하고, 타임아웃 비율을 측정

**실행**:
```bash
k6 run docs/week6/k6-scripts/scenario-timeout.js
```

---

### Scenario 2: CircuitBreaker 검증

**목적**: 실패율이 임계치(50%)를 초과하면 서킷이 Open되어 PG 호출을 차단하는지 확인

**검증 포인트**:
- 10건 슬라이딩 윈도우 내 실패율 50% 초과 시 서킷 Open
- Open 상태에서 PG 호출 없이 즉시 Fallback 응답 반환
- 10초 후 Half-Open 전환, 2건 테스트 호출 허용
- Half-Open에서 성공 시 Closed 복귀

**시나리오**: 높은 동시 요청으로 PG 실패율을 높여 서킷 상태 전이(Closed -> Open -> Half-Open -> Closed) 관찰

**실행**:
```bash
k6 run docs/week6/k6-scripts/scenario-circuit-breaker.js
```

---

### Scenario 3: Fallback 검증

**목적**: PG 장애 시 Fallback 응답이 정상 반환되어 사용자에게 적절한 안내가 가능한지 확인

**검증 포인트**:
- PG 실패/타임아웃 시 500 에러가 아닌 정상 응답(fallback) 반환
- 서킷 Open 상태에서 Fallback 응답이 빠르게 반환되는지 (PG 호출 없이)
- Fallback 응답 시 Payment 상태가 적절하게 처리되는지
- Fallback 응답 latency가 정상 응답 대비 매우 짧은지

**시나리오**: 대량 요청으로 서킷을 Open시킨 후, Fallback 응답의 비율과 응답 시간을 측정

**실행**:
```bash
k6 run docs/week6/k6-scripts/scenario-fallback.js
```

---

### Scenario 4: Retry 검증

**목적**: 일시적 PG 실패 시 재시도 정책(최대 2회, 500ms 간격)이 동작하여 성공률을 높이는지 확인

**검증 포인트**:
- 첫 요청 실패 후 재시도가 수행되는지
- 재시도 간격(500ms)이 준수되는지
- 최대 재시도 횟수(2회) 초과 시 최종 실패 처리되는지
- 재시도로 인해 전체 응답 시간이 늘어나는 정도 측정
- 재시도 가능한 예외(5xx, ResourceAccessException)만 재시도하는지

**시나리오**: 중간 수준의 부하로 PG 실패를 유발하고, 재시도 유무에 따른 최종 성공률을 비교

**실행**:
```bash
k6 run docs/week6/k6-scripts/scenario-retry.js
```

## V2 시나리오 (Resilience4j - `POST /api/v2/payments`)

V2는 Resilience4j 어노테이션 기반으로 구현되어 있으며, V1(Manual)과 다음이 다릅니다.

### V1 vs V2 비교

| 항목 | V1 (Manual) | V2 (Resilience4j) |
|------|------------|-------------------|
| Timeout | RestTemplate read 3s | Feign read 3s + **TimeLimiter 4s** (전체 상한) |
| Retry | 최대 2회, 500ms 간격 | **최대 3회** (첫 시도 + 재시도 2회), 500ms 간격 |
| CircuitBreaker | ManualCircuitBreaker 클래스 | **@CircuitBreaker** 어노테이션 |
| Fallback | 수동 try-catch | **2단계**: PgApiClient fallback + PaymentFacade try-catch |
| Slow Call 감지 | 없음 | **2초 초과 = 느린 호출**, 50% 초과 시 서킷 Open |

### Resilience4j 설정

```
적용 순서 (바깥 → 안쪽):
  Retry(3회, 500ms) → CircuitBreaker(pgCircuit) → TimeLimiter(4s) → Feign(connect 1s, read 3s)
```

```yaml
resilience4j:
  timelimiter:
    instances:
      pgTimeLimiter:
        timeout-duration: 4s
        cancel-running-future: true
  retry:
    instances:
      pgRetry:
        max-attempts: 3
        wait-duration: 500ms
        retry-exceptions:
          - feign.RetryableException
        ignore-exceptions:
          - feign.FeignException$BadRequest
          - feign.FeignException$NotFound
        fail-after-max-attempts: true
  circuitbreaker:
    instances:
      pgCircuit:
        sliding-window-size: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10s
        permitted-number-of-calls-in-half-open-state: 2
        slow-call-duration-threshold: 2s
        slow-call-rate-threshold: 50
```

### V2 Scenario 1: Timeout 검증

**목적**: Feign read timeout(3s)과 TimeLimiter(4s) 2단계 타임아웃이 정상 동작하는지 확인

**검증 포인트**:
- 개별 PG 호출이 Feign read timeout(3s) 내에 끊기는지
- Retry 포함 전체 실행이 TimeLimiter(4s) 이내로 제한되는지
- 타임아웃 시 fallback이 TimeoutException을 감지하여 UNKNOWN 상태 반환
- p99 응답 시간이 8초(TimeLimiter 4s + 여유)를 넘지 않는지

**시나리오**: 점진적 부하 증가(1→50 VUs, 60s)로 PG 지연 유발

**실행**:
```bash
k6 run -e BASE_URL=http://localhost:8080 -e MIN_ORDER_ID=1 -e MAX_ORDER_ID=100 \
  docs/week6/k6-scripts/v2-scenario-timeout.js
```

---

### V2 Scenario 2: CircuitBreaker 검증

**목적**: Resilience4j @CircuitBreaker가 실패율/느린호출 기준으로 서킷을 Open하는지 확인

**검증 포인트**:
- 실패율 50% 초과 또는 느린 호출(2초 초과) 50% 초과 시 서킷 Open
- Open 상태에서 CallNotPermittedException → 즉시 fallback (< 50ms)
- 10초 후 Half-Open 전환, 2건만 테스트 통과
- slow-call-duration-threshold(2s) 초과 호출도 실패로 카운트

**시나리오**: 고부하(60 VUs, 70s)로 서킷 상태 전이 관찰 (Closed → Open → Half-Open → Closed)

**실행**:
```bash
k6 run -e BASE_URL=http://localhost:8080 -e MIN_ORDER_ID=1 -e MAX_ORDER_ID=100 \
  docs/week6/k6-scripts/v2-scenario-circuit-breaker.js
```

---

### V2 Scenario 3: Fallback 검증

**목적**: 2단계 Fallback 구조가 어떤 장애에서도 500 에러 없이 안전한 응답을 반환하는지 확인

**검증 포인트**:
- 1단계 fallback: TimeLimiter → TIMEOUT(UNKNOWN), CircuitBreaker → FAILED, Retry 소진 → FAILED
- 2단계 fallback: PaymentFacade try-catch로 예측 불가 예외도 처리
- 전체 요청 중 5xx 에러 비율 5% 미만
- 서킷 Open fallback은 50ms 미만, 타임아웃 fallback은 2s 이상

**시나리오**: 대량 요청(80 VUs, 60s)으로 다양한 fallback 경로 유발

**실행**:
```bash
k6 run -e BASE_URL=http://localhost:8080 -e MIN_ORDER_ID=1 -e MAX_ORDER_ID=100 \
  docs/week6/k6-scripts/v2-scenario-fallback.js
```

---

### V2 Scenario 4: Retry 검증

**목적**: Resilience4j @Retry(3회, 500ms)가 성공률을 높이는지, TimeLimiter(4s)와 조합이 정상인지 확인

**검증 포인트**:
- 재시도로 최종 성공률이 PG 원래 성공률(60%)보다 높아지는지
- 응답 시간 분포로 재시도 횟수 추정:
  - 1회 시도: < 600ms
  - 2회 시도: 600~1500ms (500ms 대기 + 재시도)
  - 3회 시도: 1500ms+ (500ms × 2 대기 + 재시도)
- TimeLimiter(4s)에 의해 Retry가 중간에 끊기는 케이스 존재 확인
- 4xx 에러(BadRequest, NotFound)는 재시도 없이 즉시 실패

**시나리오**: 중간 부하(15 VUs, 60s)로 재시도 효과 측정

**실행**:
```bash
k6 run -e BASE_URL=http://localhost:8080 -e MIN_ORDER_ID=1 -e MAX_ORDER_ID=100 \
  docs/week6/k6-scripts/v2-scenario-retry.js
```

## 실행 결과 확인

### k6 출력 지표
| 지표 | 의미 |
|------|------|
| `http_req_duration` | 요청 소요 시간 (p50, p90, p95, p99) |
| `http_req_failed` | 실패율 |
| `http_reqs` | 총 요청 수 |
| `iterations` | 완료된 반복 수 |

### 커스텀 지표
| 지표 | 의미 |
|------|------|
| `payment_success` | 결제 성공 (PENDING 포함) 비율 |
| `payment_failed` | 결제 실패 비율 |
| `payment_fallback` | Fallback 응답 비율 |
| `payment_timeout` | 타임아웃 비율 |

### Grafana 대시보드
- `http://localhost:3000` (admin/admin)
- Resilience4j 메트릭 확인:
  - `resilience4j_circuitbreaker_state` - 서킷 상태
  - `resilience4j_circuitbreaker_calls_seconds` - 호출 시간
  - `resilience4j_retry_calls_total` - 재시도 횟수
