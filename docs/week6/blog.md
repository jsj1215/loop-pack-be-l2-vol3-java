## 들어가며

주문의 동시성 이슈를 해결해보고 이제는 그 주문에 PG사와 연동하는 결제 시스템을 만들 차례다. 외부 api만 연결 한다고 하면 구현 자체는 어렵지 않을 수 있지만, 문제는 **외부 api에 문제가 생겼을 때** 벌어지는 일이다.

PG 서버가 장애를 일으키면, 우리 서비스는 PG 응답을 기다리면서 멈춘다. 요청이 계속 들어오면 스레드 풀이 고갈되고, 결국 결제와 상관없는 API까지 전부 문제가 생긴다. 외부 시스템 하나가 죽었을 뿐인데, 우리 시스템 전체가 마비되는 상황이 발생한다.

이걸 어떻게 막아야 할까? 문제가 생겼다는걸 감지하고 우리 서비스에 영향을 주기전에 막을 수 있는건 없을까???

## Part 1. 서킷 브레이커란 무엇인가?

### 1. 전기 회로의 차단기에서 온 이름

Circuit Breaker는 원래 **전기 공학 용어**다. 한국어로는 "회로 차단기" 또는 "누전 차단기"라고 부른다.

집에서 에어컨, 전자레인지, 드라이기를 동시에 틀면 차단기가 내려간 경험이 있을 것이다. **전류가 과하게 흐르면 회로를 끊어서 화재가 발생할 수 있는 위험을 막는 것**이 Circuit Breaker의 본질이다.

소프트웨어에서도 똑같은 원리를 적용할 수 있다.
이해하기 쉽게 전기 회로에 비유하여 소프트웨어에서의 서킷 브레이커를 정의해보자.

| 전기 회로 | 소프트웨어 | 설명 |
|---|---|---|
| **전기 회로** | **외부 시스템 호출 경로** | 우리 서비스 → PG사 API 같은 네트워크 호출 |
| **전류** | **요청(Request)** | 회로를 통해 흐르는 것 |
| **과전류 / 단락** | **연속 실패 / 타임아웃** | 외부 시스템이 장애 상태 |
| **차단기 작동** | **서킷 Open** | 요청을 더 이상 보내지 않고 즉시 실패를 반환 |
| **화재 / 가전 고장** | **장애 전파 / 시스템 마비** | 보호하지 않으면 발생하는 피해 |

핵심은 **"부분적인 장애가 전체 시스템으로 퍼지는 걸 막는 안전장치"**라는 점이다.

### 2. 서킷 브레이커가 없으면 어떤 일이 벌어질까?

```
사용자 요청 → 우리 서비스 → PG 서버 (장애 발생, 응답 지연)
                  ↓
         스레드가 PG 응답을 무한정 대기
                  ↓
         스레드 풀 고갈 (다른 요청도 처리 불가)
                  ↓
         전체 서비스 마비
```

- 외부 시스템이 죽어도 계속 요청을 보내게 된다 → **쓸데없는 네트워크/스레드 자원 낭비**
- 응답이 올 때까지 마냥 기다린다 → **스레드 풀 고갈**
- 에러 로그가 폭발적으로 쌓인다 → **CPU 사용률 급등**

결국 PG 하나 때문에 **우리 시스템 전체가 죽는다.**

그럼 서킷 브레이커가 있으면?
- 실패가 반복되면 **아예 호출 자체를 차단** (Fail-Fast)
- 차단된 동안은 **Fallback 응답을 바로 반환** → 스레드를 잡고 있지 않음
- 주기적으로 외부 시스템이 살아났는지 확인 → **자동 복구**

### 3. 세 가지 상태와 전이

서킷 브레이커에는 세 가지의 상태가 있다.

```
                    실패율 >= 임계치
         ┌──────────────────────────────┐
         │                              ▼
     ┌────────┐                    ┌────────┐
     │ CLOSED │                    │  OPEN  │
     │ (정상)  │                    │ (차단)  │
     └────────┘                    └────────┘
         ▲                              │
         │       wait-duration 경과       │
         │                              ▼
         │                        ┌───────────┐
         │    시험 성공            │ HALF-OPEN │
         └────────────────────────│ (반개방)   │
                                  └───────────┘
                                        │
                                        │ 시험 실패
                                        ▼
                                   ┌────────┐
                                   │  OPEN  │
                                   └────────┘
```

**Closed (정상)** — 차단기가 닫혀 있어서 전류(요청)가 정상적으로 흐르는 상태. 요청마다 성공/실패를 슬라이딩 윈도우에 기록하고, 실패율이 임계치를 넘으면 Open으로 바뀐다.

**Open (차단)** — 차단기가 열려서 전류(요청)가 안 흐르는 상태. 요청을 보내지 않고 바로 `CallNotPermittedException`을 던지거나 Fallback을 실행한다. 설정해둔 대기 시간이 지나면 Half-Open으로 넘어간다.

**Half-Open (시험)** — 차단기를 살짝 올려서 전류를 조금만 흘려보는 것. 설정된 수만큼 시험 요청을 보낸 뒤, 전체 실패율이 임계치 미만이면 Closed로, 이상이면 다시 Open으로 간다. 주의할 점은 시험 요청이 **모두 완료된 후** 실패율을 계산한다는 것이다 — 도중에 성공이 많다고 바로 전환되지 않는다.

---

서킷브레이커의 세가지 상태를 결정 하기 위해 요청 상태가 정상인지 실패인지를 슬라이딩 윈도우에 기록한다고 했다.
그렇다면 슬라이딩 윈도우는 어떤 것일까?

### 4. 슬라이딩 윈도우 — "실패율은 어떻게 판단할까?"

서킷 브레이커가 "실패율"을 판단하려면 최근 요청들의 성공/실패를 추적해야 한다.

> 기차를 타고 가면서 창문 밖을 바라본다고 생각해보자.
> 창문(Window)의 크기는 고정인데, 기차가 이동(Slide)하면서 보이는 풍경은 계속 바뀐다.
> 지나간 풍경은 창문 밖으로 사라지고, 새로운 풍경이 들어온다.

핵심은 전체 이력이 아니라 **"최근 상태"**만 보고 판단한다는 것이다.

#### 왜 고정 윈도우가 아니라 슬라이딩 윈도우일까?

고정 윈도우(Fixed Window)는 정해진 구간(예: 매 1분)마다 집계를 리셋한다. 문제는 **경계 효과**다.

```
|---- 1분 구간 ----|---- 1분 구간 ----|
................XXXXX|XXXXX...........
                    ↑
        이 10초 동안 실패가 집중되었는데,
        두 구간에 나뉘어서 각각은 정해진 임계치에 미달
```

구간 경계에 걸쳐서 실패가 몰리면 감지를 못 하는 **사각지대**가 생긴다. 슬라이딩 윈도우는 관찰 범위가 계속 이동하니까 이 경계 효과를 **크게 줄일 수 있다**. 다만 Time-based 슬라이딩 윈도우는 내부적으로 부분 집계 버킷을 쓰기 때문에 버킷 단위의 미세한 경계 효과가 남아있을 수 있다. Count-based는 호출 단위로 링 버퍼에 기록하니까 이 문제에서 자유롭다 — 이것도 COUNT_BASED를 선택한 이유 중 하나다.

#### Count-based vs Time-based

Resilience4j는 두 가지 슬라이딩 윈도우를 제공한다.

- **Count-based**: 최근 N**개** 호출을 링 버퍼에 기록. 고정 크기 배열이라 O(1)로 동작한다.
- **Time-based**: 최근 N**초** 동안의 호출을 부분 집계 버킷에 기록.

**결론: `COUNT_BASED`를 선택했다.** PG 결제는 사용자가 결제 버튼을 눌러야 발생하니까 트래픽이 산발적이다. Time-based를 쓰면 트래픽이 적은 시간대에 윈도우 안에 데이터가 부족해서 오판할 수 있다.

## Part 2. 설정

### Resilience4j

**Resilience4j**는 Java용 경량 장애 허용(fault tolerance) 라이브러리다.
Circuit Breaker, Retry, Rate Limiter, TimeLimiter 같은 패턴을 어노테이션 하나로 간결하게 적용할 수 있고, Spring Boot랑 궁합이 좋아서 YAML 설정만으로 세밀하게 설정할 수 있다.
사실상 Java 생태계의 표준이고, 함수형 프로그래밍 기반 설계라 가볍고 의존성이 적다는 점에서 선택했다.

### 전체 설정과 실행 순서

```yaml
resilience4j:
  circuitbreaker:
    instances:
      pgCircuit:
        sliding-window-size: 10                          # 최근 10건의 호출 결과로 실패율 판단
        minimum-number-of-calls: 10                      # 최소 10건이 쌓여야 실패율 계산 시작
        failure-rate-threshold: 50                        # 실패율 50% 이상이면 서킷 OPEN
        wait-duration-in-open-state: 10s                  # OPEN 후 10초 대기 → HALF_OPEN 전이
        permitted-number-of-calls-in-half-open-state: 3   # HALF_OPEN에서 3건 시험 후 판정
        slow-call-duration-threshold: 3s                  # 3초 초과 응답을 "느린 호출"로 분류
        slow-call-rate-threshold: 50                      # 느린 호출이 50% 넘으면 서킷 OPEN

  # Retry: PG의 일시적 장애(네트워크 순단, 서버 재시작 등)는
  # 잠깐 기다렸다가 다시 시도하면 성공할 수 있다.
  # 서킷 브레이커가 "반복 실패 시 차단"이라면, Retry는 "한두 번 실패는 다시 시도"하는 역할이다.
  retry:
    instances:
      pgRetry:
        max-attempts: 3                # 최대 3회 시도 (첫 시도 + 재시도 2회)
        wait-duration: 500ms           # 재시도 간격 500ms
        retry-exceptions:              # 아래 예외만 재시도 대상
          - feign.RetryableException
          - feign.FeignException$InternalServerError
          - feign.FeignException$BadGateway
          - feign.FeignException$ServiceUnavailable
          - feign.FeignException$GatewayTimeout
        fail-after-max-attempts: true  # 소진 후 예외를 던져 Fallback 트리거

  # TimeLimiter: PG가 응답을 안 주면 스레드가 영원히 멈출 수 있다.
  # Feign의 read timeout(3s)과 별개로, Future 자체에 시간 상한을 걸어서
  # 어떤 상황에서도 무한 대기가 발생하지 않도록 하는 최후의 안전망이다.
  timelimiter:
    instances:
      pgTimeLimiter:
        timeout-duration: 4s             # 개별 호출(Future) 4초 초과 시 타임아웃
        cancel-running-future: true      # 타임아웃 시 실행 중인 Future 취소
```

다만 `cancel-running-future: true`는 **Java의 Future를 취소**하는 것이지, 이미 네트워크로 나간 HTTP 요청 자체를 끊는 건 아니다. 실제로 PG에 도달한 요청은 PG 쪽에서 처리가 끝날 수 있고, 우리 쪽에서는 그 응답을 버리는 것이다.

서킷 브레이커 외에 Retry(3회, 500ms 간격)와 TimeLimiter(4초)도 함께 적용했다.
Resilience4j는 데코레이터 패턴으로 이들을 조합하며, 기본 order 값에 따라 **어노테이션 순서와 무관하게** 항상 같은 순서로 실행된다.(중요!!)

```
Retry(3회) → CircuitBreaker → TimeLimiter(4s) → Feign HTTP(3s)
```

여기서 주의할 점은 **TimeLimiter가 Retry 안쪽에 있다**는 것이다. TimeLimiter는 Future 단위로 적용되는데, Retry가 각 시도마다 새로운 호출(새로운 Future)을 수행하므로 TimeLimiter도 **각 시도마다 별도로** 적용된다. Retry를 포함한 전체 시간에 상한을 거는 게 아니다. 따라서 현재 설정에서는 최대 4s × 3회 + 대기 500ms × 2회 = **약 13초**까지 소요될 수 있다. 전체 요청 시간에 상한을 두려면 aspect order를 명시적으로 설정해야 한다.

서킷이 Open일 때의 흐름을 구체적으로 보면 이렇다. 가장 바깥의 Retry가 CircuitBreaker를 호출하면, CircuitBreaker는 "나 Open이야!" 하고 `CallNotPermittedException`을 던진다. Retry가 이 예외를 받지만, 위의 `retry-exceptions` 목록에 `CallNotPermittedException`이 포함되어 있지 않으므로 재시도 없이 즉시 포기한다. 즉, **Retry가 서킷을 감싸고 있지만 서킷의 "차단 예외"는 재시도 대상에서 제외했기 때문에 불필요한 재시도 없이 Fail-Fast**가 일어난다.

### 설정별 선택 근거

각 설정값에 대해 "왜 이 숫자인지"를 선택했는지 하나씩 살펴보자.

**1.`sliding-window-size: 10` / `minimum-number-of-calls: 10`**

최근 10개의 요청 결과를 보고 실패율을 판단한다. 최소 10건이 쌓여야 판단을 시작하니까, 서비스를 막 시작했을 때 한두 건 실패했다고 바로 차단되는 일은 없다.

10이라는 숫자는 작은 편이다. 실제로 테스트해보니 PG 실패율이 40%인 상황에서 10건 중 5건 이상 실패할 확률이 약 36.7%나 되어서, **서킷이 잠깐 열렸다가 닫히고, 또 열렸다가 닫히는 현상**이 관찰됐다. 진짜 장애가 아닌데도 실패가 우연히 몰리면 서킷이 열리고, 다음 10건에서는 괜찮아서 다시 닫히는 식이다. 다만 실제로 차단된 요청은 전체의 0.6% 수준이라 서비스에 큰 영향은 없었다. 더 안정적으로 만들고 싶다면 20으로 올리는 것을 고려할 수 있다. 다만 윈도우가 커지면 그만큼 장애 감지가 늦어질 수 있다는 트레이드오프가 있다.

> **포기한 것**: 윈도우가 10이라 서킷이 열렸다 닫혔다를 반복한다. PG가 실제로는 60% 성공시키고 있어도, 운 나쁘게 실패가 몰리면 서킷이 열려서 정상 요청까지 차단된다. 반대로 100으로 올리면 50건이 터지는 동안 계속 PG에 요청을 보내게 된다.

**2.`failure-rate-threshold: 50`**

10건 중 5건 이상 실패하면 서킷을 연다. PG의 평균 실패율(40%)보다 10%p 높게 잡았다. 40%로 낮추면 정상 상황에서도 서킷이 자주 열리고, 60%로 높이면 실제 장애를 놓칠 수 있다. 50%는 "평소에는 안 열리지만, 진짜 문제가 생기면 반응하는" 적절한 경계선이라고 생각했다.

> **포기한 것**: PG가 실패율 45%로 느리게 악화되는 장애는 감지하지 못한다. 50% 미만이면 서킷이 안 열리기 때문이다. 반면 70%로 올리면 진짜 장애 시에도 70%까지 실패가 쌓여야 차단이 시작되어, 그 사이에 들어온 요청은 전부 PG에 보내진다.

**3.`wait-duration-in-open-state: 10s`**

서킷이 열리면 10초 동안 PG에 요청을 보내지 않는다. 너무 짧으면 아직 복구 안 된 PG에 요청을 보내서 또 실패하고, 너무 길면 PG가 살아났는데도 오래 차단한다. 테스트에서 복구까지 약 15초가 걸렸는데, 이론적 복구 시간(약 11초: OPEN 대기 10초 + HALF_OPEN 프로브 3건)에 가까운 수치여서 10초가 적절했다.

> **포기한 것**: OPEN 상태 10초 동안 들어오는 모든 결제 요청은, PG가 50% 성공시키고 있었더라도 시도조차 하지 않고 즉시 실패 처리된다. 5초로 줄이면 아직 복구 안 된 PG에 프로브를 보내서 HALF_OPEN → OPEN 사이클만 반복할 수 있다.

**4.`permitted-number-of-calls-in-half-open-state: 3`**

10초 후 '하프-오픈' 상태에서 3건만 시험적으로 보내본다. 3건이 **모두 완료된 후** 실패율을 계산하여 `failure-rate-threshold(50%)` 미만이면 CLOSED로 복귀하고, 이상이면 다시 OPEN으로 돌아간다. 예를 들어 2건 성공 + 1건 실패 = 실패율 33.3% < 50%이면 CLOSED다. 1건이면 운이 나빠 실패할 수 있고, 너무 많으면 아직 불안정한 PG에 부담을 줄 수 있어 3건을 선택했다.

> **포기한 것**: 3건은 표본이 적어서 운에 좌우될 수 있다. PG가 복구됐는데 3건 중 2건이 운 나쁘게 실패하면 다시 OPEN으로 돌아가고, 또 10초를 기다려야 한다. 반대로 10건으로 늘리면 아직 불안정한 PG에 10건을 보내야 판정이 끝나서, 복구 시점의 부담이 커진다.

**`slow-call-duration-threshold: 3s` / `slow-call-rate-threshold: 50`**

응답이 3초 넘게 걸리는 요청을 "느린 호출"로 분류하고, 이게 50%를 넘으면 서킷을 연다. 실패뿐 아니라 "느려진 것"도 장애의 신호로 본다는 뜻이다. 다만 현재 PG는 빠르게 응답(100~500ms)하므로 이 설정이 실제로 동작한 적은 없었다. PG가 진짜 느려지는 장애가 발생할 때를 위한 **안전망**이다.

> **포기한 것**: 3초 기준이라 2.9초짜리 응답은 "정상"으로 분류된다. PG가 점진적으로 느려지는 경우(500ms → 1s → 2s) 3초에 도달하기 전까지는 감지하지 못한다. 1초로 낮추면 PG 부하 시 정상 응답(700ms~1s)도 "느린 호출"로 잡힐 수 있다.

### 코드: 어노테이션과 Fallback 구현

Resilience4j의 장점은 어노테이션 세 개만 붙이면 위의 모든 설정이 적용된다는 점이다.

```java
@TimeLimiter(name = "pgTimeLimiter")
@CircuitBreaker(name = "pgCircuit")
@Retry(name = "pgRetry", fallbackMethod = "requestPaymentFallback")
public CompletableFuture<PgPaymentResponse> requestPayment(Long memberId, PgPaymentRequest request) {
    return CompletableFuture.supplyAsync(() ->
            pgFeignClient.requestPayment(String.valueOf(memberId), request), pgExecutor);
}
```

이 한 메서드에 세 가지 보호막이 겹겹이 감싸져 있다. 실행 순서는 `Retry → CircuitBreaker → TimeLimiter → 실제 PG 호출`이다. 서킷이 열려 있으면 안쪽의 TimeLimiter와 PG 호출은 아예 실행되지 않고 즉시 차단된다.

모든 실패는 결국 **Fallback 메서드**로 흘러간다.

```java
private CompletableFuture<PgPaymentResponse> requestPaymentFallback(
        Long memberId, PgPaymentRequest request, Throwable t) {
    if (t instanceof TimeoutException) {
        return CompletableFuture.completedFuture(PgPaymentResponse.timeout());
    }
    return CompletableFuture.completedFuture(
            PgPaymentResponse.failed("PG 시스템 장애: " + t.getMessage()));
}
```

타임아웃이면 "아직 모르겠다(UNKNOWN)", 그 외 에러면 "실패(FAILED)"라는 의미 있는 응답을 돌려준다. 사용자에게 500 에러 화면을 보여주는 대신, "결제가 실패했습니다. 다시 시도해주세요"라는 안내를 할 수 있게 되는 것이다.

### 서킷이 차단한 요청 — 진짜 실패일까?

서킷이 OPEN 상태에서 Fallback을 실행하면, PG에 요청 자체를 보내지 않는다. 이건 "PG가 에러를 반환한 것"과 본질적으로 다르다.

```
[PG 에러로 인한 실패]
  우리 서비스 ────→ PG ────→ "500 에러"  → 확실한 실패 ✓

[서킷 차단으로 인한 실패]
  우리 서비스 ──X (서킷 OPEN, PG 호출 안 함)  → PG가 성공시켰을 수도 있음 ⚠
```

서킷이 차단한 요청은 **"실패한 게 아니라 시도하지 않은 것"**이다. PG가 50% 성공시키고 있는 상황에서 서킷이 열리면, 성공했을 수도 있는 요청까지 차단된다. 이건 서킷 브레이커가 **의도적으로 감수하는 트레이드오프**다 — 나머지 50%의 실패가 우리 시스템까지 끌어내리는 것보다, 100%를 빠르게 차단하고 시스템을 보호하는 쪽을 택한 것이다.

Fallback에서 이 두 가지를 구분하는 것도 중요하다.

```
Fallback 실행
    │
    ├─ CallNotPermittedException (서킷 차단)
    │   → "시도하지 않음" — 잠시 후 재시도 안내
    │
    ├─ TimeoutException (TimeLimiter 초과)
    │   → "결과를 모름" — PG가 성공했을 수 있으므로 PENDING 유지
    │
    └─ 그 외 예외 (PG 에러 + Retry 소진)
        → "확실한 실패" — FAILED 처리
```

서킷 차단과 타임아웃은 모두 Fallback으로 흘러오지만, **"시도하지 않은 것"과 "시도했지만 결과를 모르는 것"은 다르다.** 서킷 차단은 안전하게 재시도할 수 있지만, 타임아웃은 PG가 이미 처리했을 수 있어서 함부로 재시도하면 중복 결제가 발생할 수 있다. 결국 서킷 브레이커의 Fallback은 단순히 "에러 메시지를 예쁘게 포장하는 것"이 아니라, **차단 원인에 따라 후속 처리를 달리해야 하는 분기점**이다.

## Part 3. 이 설정이 최선일까? — k6를 활용한 부하 테스트

설정값을 정하고 코드를 짰으니, 이제 "진짜 이게 동작하는가?"를 확인해봐야 한다. k6로 가상 사용자(VU) 5~120명을 만들어 결제 요청을 시도했다.

### 테스트 환경

PG 시뮬레이터를 띄워서 실제 PG사처럼 동작하게 했다.

- **요청 성공 확률**: 60% (40%는 서버 에러를 반환)
- **응답 시간**: 100ms ~ 500ms
- **테스트 데이터**: 주문 10,000건, 가상 사용자마다 고유한 주문 ID 할당

각 패턴이 제대로 동작하는지 개별 시나리오로 먼저 확인하고, 마지막에 모든 패턴을 동시에 동작시키는 종합 테스트를 돌렸다. 시나리오별로 VU 수와 테스트 시간이 다르기 때문에 요청 수도 다르다.

| 시나리오 | 검증 대상 | VU 수 | 요청 수 | 핵심 결과 |
|---------|----------|-------|--------|----------|
| 타임아웃 | TimeLimiter 4s 상한 | 5→50 | 2,166 | 3초 초과 응답 **0건** |
| 서킷 브레이커 | 서킷 OPEN 전이 | 3→60 | 2,801 | fast-fail **51건**, 5xx **0건** |
| Fallback | 장애 시 응답 커버리지 | 10→100 | 7,691 | 5xx **0건**, 의미 있는 응답(성공 또는 Fallback) **97.77%** |

- **타임아웃**: PG Simulator의 응답이 100~500ms로 빨라서 타임아웃에 도달하지는 않았지만, TimeLimiter 4초 상한이 설정되어 있으므로 PG가 진짜 느려져도 무한 대기는 발생하지 않는다는 안전망이 확인됐다.
- **서킷 브레이커**: 51건의 응답이 100ms 미만에 돌아왔다. 정상적인 PG 호출은 최소 100ms 이상 걸리므로, 100ms 미만이라는 건 **PG에 아예 요청을 보내지 않고 즉시 차단**했다는 증거다.
- **Fallback**: 7,691건 중 5xx 에러가 단 한 건도 없었다. PG가 에러를 뱉든, 서킷이 차단하든, 사용자에게는 항상 "결제 실패" 또는 "잠시 후 재시도"라는 의미 있는 응답이 내려갔다.

### 종합 테스트 — 모든 패턴을 동시에 동작시키면?

가상 사용자 5명에서 120명까지, 총 **7,259건**의 요청을 쏟아부었다.

| 평가 항목 | 결과 | 한 줄 요약 |
|----------|------|-----------|
| 5xx 에러 | **0건** | 어떤 상황에서도 서버 에러가 사용자에게 노출되지 않았다 |
| 응답 성공률 (Fallback 포함) | **99.51%** | HTTP 200 + 유효한 결제 상태를 반환한 비율. 나머지 0.49%는 orderId 충돌로 인한 비즈니스 에러 |
| 서킷 차단 | **459건 (6.3%)** | 불필요한 PG 호출을 막아 자원을 아꼈다 |
| Fallback 커버리지 | **100%** | 모든 장애 케이스에서 Fallback이 정상 동작했다 |

Fallback이 커버한 장애 유형을 분류해보면:

| 장애 유형 | Fallback 동작 | 사용자에게 전달된 응답 |
|----------|--------------|---------------------|
| PG 에러 + Retry 소진 | 3회 시도 후 Fallback 실행 | "결제에 실패했습니다" (FAILED) |
| 서킷 OPEN | PG 호출 없이 즉시 Fallback | "잠시 후 다시 시도해주세요" (FAILED) |

PG가 에러를 뱉든, 서킷이 차단하든, 500 에러 화면 대신 **항상 의미 있는 응답**이 내려갔다. TimeLimiter + Retry + CircuitBreaker + Fallback 모두 각자의 역할을 한 것이다.

- **TimeLimiter**: "개별 호출이 4초를 넘으면 끊어" → 무한 대기 방지 (Retry 포함 시 최대 13초)
- **Retry**: "PG가 일시적으로 실패하면 500ms 후에 다시 시도해" → 일시적 장애 극복
- **CircuitBreaker**: "실패가 계속되면 아예 요청을 보내지 마" → 장애 확산 차단
- **Fallback**: "어떤 이유로든 실패하면, 사용자에게 의미 있는 응답을 줘" → 사용자 경험 보호

## 서킷 브레이커 설정 트레이드오프 총정리

서킷 브레이커의 각 설정값은 **"빨리 감지할 것인가 vs 잘못된 판단을 줄일 것인가"**라는 하나의 축 위에서 균형을 잡는 것이다. 한눈에 정리하면 이렇다.

| 설정 | 얻은 것 | 포기한 것 |
|-----|--------|----------|
| `sliding-window-size: 10` | 장애 감지가 빠름 (10건이면 판정) | 서킷이 열렸다 닫혔다 반복 — 정상 상황에서도 실패가 몰리면 서킷이 열림 |
| `failure-rate-threshold: 50` | 평소에는 안 열리지만 장애 시 반응 | 실패율 45% 수준의 느린 악화는 감지 못함 |
| `wait-duration: 10s` | PG 복구 시 빠르게 재개 | 10초간 PG가 50% 성공시켜도 전부 차단 |
| `half-open: 3건` | 불안정한 PG에 최소한의 부담 | 3건은 운에 좌우 — PG가 복구됐는데 다시 OPEN될 수 있음 |
| `slow-call: 3s / 50%` | 지연도 장애로 감지 | 2.9초짜리 점진적 악화는 감지 못함 |

```
          빠른 감지 ◀──────────────────────▶ 적은 오판
                    │                     │
  window 작게 (10)  │                     │  window 크게 (100)
  threshold 낮게 (50%) │                  │  threshold 높게 (70%)
  wait 짧게 (5s)    │                     │  wait 길게 (30s)
                    │                     │
                    │    현재 설정 (★)      │
                    │     window=10        │
                    │     threshold=50%    │
                    │     wait=10s         │
```

현재 설정은 **빠른 감지 쪽에 가깝다.** 그래서 테스트에서 서킷이 열렸다 닫혔다를 반복하는 현상이 관찰됐고, 서킷이 열릴 때 정상 요청도 함께 차단됐다. 하지만 실제로 차단된 요청은 전체의 0.6~6.3% 수준이었고, 5xx 에러는 0건이었다. **"가끔 과잉 차단하더라도, 장애가 퍼지는 건 확실히 막겠다"**는 나름의 전략이다.

## 마치며

서킷 브레이커를 공부할 때는 개념 자체는 이해가 됐다. "실패가 많으면 차단하고, 괜찮아지면 다시 열어주는 거구나." 근데 막상 직접 설정하고 테스트해보니까, 개념을 아는 것과 실제로 동작하게 만드는 건 완전히 다른 문제였다.

설정값을 정할 때마다 "이 숫자를 왜 이렇게 잡아야 하지?"라는 질문이 계속 나왔고, 테스트를 돌릴 때마다 예상과 다른 결과가 나와서 다시 원인을 찾아봐야 했다. 특히 60 VUs에서 서킷이 안 열린 걸 보고 "왜지?" 하면서 Retry와 서킷의 관계를 알게 된 건, 문서만 읽어서는 절대 못 얻었을 경험이었다.

결국 이번에 가장 크게 느낀 건, **설정값을 채울 때 "왜 이 값인지"를 설명할 수 있어야 한다**는 것이다. 근거 없이 기본값을 그냥 쓰면, 나중에 문제가 생겼을 때 뭘 바꿔야 할지조차 모르게 된다. 근거를 붙여두면 상황이 달라졌을 때 "그때는 이래서 이 값이었는데, 지금은 다르니까 바꿔야겠다"고 판단할 수 있다.

설정에 근거를 붙이고, 테스트로 증명하고, 결과로 판단을 수정하는 것 — 이 사이클을 돌려보면서 어떤 결정을 할 때에는 가설을 세우고 집요하게 검증을 한 결과로 선택하는 습관을 들여야 겠다고 생각했다.
